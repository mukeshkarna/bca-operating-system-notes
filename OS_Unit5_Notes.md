# PART A — BASIC MEMORY MANAGEMENT (3 Hours)

## 1. Introduction to Memory Management

**Memory Management** is the OS function that manages the computer's primary memory (RAM). It decides:
- Which processes get memory and how much.
- Where in memory each process is loaded.
- What to do when memory is full.

**Goals of Memory Management:**
- Maximize the number of processes in memory (increase CPU utilization).
- Ensure isolation — one process cannot access another's memory.
- Allow relocation — process can be placed at any memory address.
- Provide protection against unauthorized access.

> **Real-world example:** Your laptop has 8GB RAM. When you open Chrome, Word, and Spotify simultaneously, the OS memory manager decides how to distribute that 8GB among all running processes.

---

## 2. Memory Hierarchy

Memory is organized in a **hierarchy** based on speed, cost, and capacity:

```
┌─────────────────────────────────────────────────────────┐
│                    CPU Registers                         │  Fastest, Smallest, Most Expensive
├─────────────────────────────────────────────────────────┤
│                    Cache Memory                          │  L1, L2, L3
├─────────────────────────────────────────────────────────┤
│                    Main Memory (RAM)                     │  Primary Storage
├─────────────────────────────────────────────────────────┤
│                    Magnetic Disks (HDD/SSD)              │  Secondary Storage
├─────────────────────────────────────────────────────────┤
│                    Magnetic Tape                         │  Slowest, Largest, Cheapest
└─────────────────────────────────────────────────────────┘
          Speed decreases ↓        Capacity increases ↓
```

### 2.1 Registers
- Fastest storage, directly inside CPU.
- Very small — handful of 64-bit registers.
- Holds data currently being processed.
- Access time: < 1 nanosecond.

### 2.2 Cache Memory
- High-speed buffer between CPU and RAM.
- **L1 cache:** Smallest, fastest (inside CPU core, ~32KB).
- **L2 cache:** Larger, slightly slower (~256KB).
- **L3 cache:** Shared across cores (~8MB).
- Stores frequently accessed data to avoid slow RAM access.

### 2.3 Main Memory (RAM)
- Primary working memory — programs run here.
- Volatile — data lost on power off.
- Modern systems: 8GB–64GB.
- Access time: ~100 nanoseconds.

### 2.4 Magnetic Disks (HDD/SSD)
- Non-volatile secondary storage.
- Programs stored here when not running.
- Much slower than RAM but much larger.
- HDD: milliseconds access; SSD: microseconds.

### 2.5 Magnetic Tape
- Slowest, cheapest per GB.
- Sequential access only.
- Used for backups and archiving.

> **Real-world analogy:** Registers = your working desk surface (tiny but instant access). Cache = your desk drawer. RAM = your bookshelf. Disk = filing cabinet. Tape = warehouse archive.

## 3. Logical vs Physical Address Space

### 3.1 Logical Address (Virtual Address)
- Address **generated by the CPU** during program execution.
- Also called **virtual address**.
- Seen from the process's perspective — each process thinks it has its own memory starting at address 0.
- **Example:** A program reads from address 1000 — this is the logical address.

### 3.2 Physical Address
- **Actual address in RAM** — where data really resides.
- Generated by the **Memory Management Unit (MMU)**.
- The OS and hardware translate logical → physical addresses.
- **Example:** The OS maps logical address 1000 to physical address 5000 in RAM.

```
Process sees:                  RAM contains:
┌───────────┐    MMU          ┌───────────┐
│  0        │ ──translation──►│  4000     │
│  1000     │                 │  5000     │  ← logical 1000 = physical 5000
│  2000     │                 │  6000     │
└───────────┘                 └───────────┘
 Logical Address               Physical Address
```

**Key difference:**

| Logical Address | Physical Address |
|---|---|
| Generated by CPU/program | Actual location in RAM |
| Virtual — may not match RAM | Real hardware address |
| Same for all processes (0 to max) | Unique in the system |
| Used in source code and compilation | Used by memory hardware |

> **Real-world example:** Think of logical address as a seat number on a ticket (B-12), and physical address as the actual physical seat in the stadium. The usher (MMU) translates your ticket number to the real seat.

## 4. Memory Management with Swapping

**Swapping** = moving an entire process between **RAM and disk** (swap space) to free up memory for other processes.

```
                    RAM                       Disk (Swap Space)
┌─────────────────────┐           ┌──────────────────────────┐
│  Process A (Active) │           │  Process C (Swapped Out) │
│  Process B (Active) │ ◄── swap ►│  Process D (Swapped Out) │
└─────────────────────┘           └──────────────────────────┘
```

**Process:**
1. Process C needs to run but RAM is full.
2. OS **swaps out** Process B to disk.
3. OS **swaps in** Process C from disk to RAM.
4. Process C runs.

**Advantages:**
- Allows more processes than RAM can hold.
- Increases CPU utilization.

**Disadvantages:**
- Slow — disk I/O is much slower than RAM.
- Process must be swapped in completely before running.

> **Real-world example:** Virtual memory in Windows — when RAM is full, Windows moves idle program data to pagefile.sys on disk. You notice the system slows down — that's swapping overhead.

## 5. Memory Management with Bitmaps

A **bitmap** divides memory into fixed-size units (e.g., 4KB each) and uses a bit to track each unit:
- **0** = unit is free
- **1** = unit is occupied

```
Memory divided into 8 units:
┌───┬───┬───┬───┬───┬───┬───┬───┐
│ 1 │ 1 │ 0 │ 1 │ 0 │ 0 │ 1 │ 0 │   Bitmap
└───┴───┴───┴───┴───┴───┴───┴───┘
  0   1   2   3   4   5   6   7      Unit numbers

Units 0,1,3,6 = occupied (processes)
Units 2,4,5,7 = free
```

**Advantages:**
- Simple to implement.
- Easy to find free blocks (scan bitmap).

**Disadvantages:**
- Finding a run of contiguous free blocks is slow.
- Bitmap itself takes memory (1 bit per unit).

## 6. Memory Management with Linked Lists

OS maintains a **linked list** of memory segments, each node describing a hole (free) or process (occupied):

```
┌─────┬──────┬──────┬──────┐   ┌─────┬──────┬──────┬──────┐
│ P   │ start│ size │  *──►│   │ H   │  5   │  3   │  *──►│ ...
│     │  0   │  5   │      │   │(free)       │      │      │
└─────┴──────┴──────┴──────┘   └─────┴──────┴──────┴──────┘
  (Process at 0, size 5)          (Hole at 5, size 3)

P = Process segment (occupied)
H = Hole segment (free)
```

**Advantages:**
- Easy to merge adjacent free holes (coalescing).
- Efficient for variable-size allocations.

**Disadvantages:**
- Searching for free space takes O(n) time.
- Pointer overhead per segment.

## 7. Mono-programming (Single Process)

**Mono-programming** = **only one process** runs in memory at a time.

```
┌──────────────────────────────┐
│   Operating System           │  (Reserved)
├──────────────────────────────┤
│                              │
│   Single User Process        │  ← Entire remaining memory
│                              │
└──────────────────────────────┘
```

**How it works:**
- Process loads into memory, runs to completion.
- Next process loads only after current finishes.
- Used in early computers and embedded systems (MS-DOS).

**Advantages:** Simple, no memory management needed.
**Disadvantages:** CPU idle while process does I/O. Poor utilization.

---

## 8. Multi-programming

**Multi-programming** = **multiple processes** in memory simultaneously. CPU switches between them.

### 8.1 Dedicate Partitions for Each Process (Absolute Translation)
- Memory divided into **fixed partitions**.
- Each process gets one partition.
- Program compiled for specific memory address (absolute).

```
┌──────────────┐  Addr 0
│     OS       │
├──────────────┤  Addr 100K
│  Process A   │  Partition 1
├──────────────┤  Addr 200K
│  Process B   │  Partition 2
├──────────────┤  Addr 300K
│  Process C   │  Partition 3
└──────────────┘  Addr 400K
```

**Problem:** Program must know its memory location at compile time. Cannot be moved.

### 8.2 Maintaining a Single Queue (Re-locatable Translation)
- Programs compiled with **relative addresses**.
- Can be loaded at any available partition.
- OS uses a **base register** to relocate addresses at runtime.
- Single queue of ready processes — loaded into any available partition.

**Advantage:** More flexible — any process can run in any partition.

---

## 9. Mono-programming vs Multi-programming

| Feature | Mono-programming | Multi-programming |
|---|---|---|
| Processes in memory | 1 | Multiple |
| CPU utilization | Low (idle during I/O) | High |
| Memory needed | Entire memory for one process | Shared among processes |
| Complexity | Very simple | Complex (scheduling, protection) |
| Throughput | Low | High |
| Example | MS-DOS | Windows, Linux, macOS |

---

## 10. Modeling Multiprogramming

If a process spends fraction **p** of time waiting for I/O, and there are **n** processes in memory:

```
CPU Utilization = 1 − p^n

Where:
  p = fraction of time process waits for I/O (I/O bound)
  n = degree of multiprogramming (number of processes)
```

**Example:**
```
If p = 0.8 (process waits for I/O 80% of time):

n=1: CPU utilization = 1 − 0.8^1 = 20%
n=2: CPU utilization = 1 − 0.8^2 = 36%
n=4: CPU utilization = 1 − 0.8^4 = 59%
n=8: CPU utilization = 1 − 0.8^8 = 83%
```

> More processes in memory → higher CPU utilization. This is why multi-programming exists!

## 11. Multiprogramming with Fixed and Variable Partitions

### 11.1 Fixed (Static) Partitioning
- Memory divided into **fixed-size partitions** at system boot.
- Each partition holds exactly one process.
- Process size ≤ partition size.

```
┌──────────────────┐
│      OS          │  Reserved
├──────────────────┤
│   Partition 1    │  4 MB
│   (Process A)    │
├──────────────────┤
│   Partition 2    │  4 MB
│   (Process B)    │
├──────────────────┤
│   Partition 3    │  8 MB
│   (Process C)    │
└──────────────────┘
```

**Advantages:**
- Simple to implement.
- Fast allocation — no searching needed.

**Disadvantages:**
- **Internal fragmentation** — process smaller than partition wastes space.
- Partition size limits process size.
- Number of partitions fixed = maximum concurrent processes fixed.

### 11.2 Variable (Dynamic) Partitioning
- Partitions created **dynamically** based on process size.
- No fixed sizes — each process gets exactly as much as it needs.

```
Initially:                After loading A(40K), B(30K), C(50K):
┌────────────────┐        ┌──────────────┐
│                │        │  Process A   │  40K
│                │        ├──────────────┤
│  Free (128K)   │        │  Process B   │  30K
│                │        ├──────────────┤
│                │        │  Process C   │  50K
└────────────────┘        ├──────────────┤
                          │  Free        │  8K
                          └──────────────┘
```

**Advantages:**
- No internal fragmentation.
- Process gets exactly the memory it needs.

**Disadvantages:**
- **External fragmentation** — free space scattered in small holes.
- Allocation and deallocation is slower.

## 12. Relocation and Protection

### 12.1 Relocation
The ability to load a process at **any available memory location** rather than a fixed one.

**How it works:**
- **Base Register:** Holds the starting physical address of a process.
- **Limit Register:** Holds the size of the process.
- Physical Address = Base Register + Logical Address.

```
Logical Address: 500
Base Register:   4000
Physical Address: 4000 + 500 = 4500

If Logical Address > Limit Register → Memory Violation!
```

### 12.2 Protection
Prevents a process from accessing another process's memory.

**Mechanism:**
- Hardware checks every memory access: `0 ≤ Logical Address < Limit`
- If violated → **segmentation fault / memory access violation**.
- OS cannot be corrupted by user processes.

> **Real-world example:** When a C program has a null pointer dereference, the OS catches the illegal memory access and sends a SIGSEGV (segmentation fault) signal — this is memory protection in action.

---

## 13. Fragmentation

**Fragmentation** = wasted memory that cannot be effectively used.

### 13.1 Internal Fragmentation
- Wasted space **inside** an allocated partition.
- Occurs in **fixed partitioning**.
- Process smaller than partition → unused space inside partition.

```
Partition size:  8MB
Process size:    5MB
Wasted (internal fragmentation): 3MB
┌────────────────────┐
│   Process (5MB)    │
├────────────────────┤
│   WASTED (3MB)     │  ← Internal Fragmentation
└────────────────────┘
```

### 13.2 External Fragmentation
- Wasted space **outside** allocated partitions.
- Occurs in **dynamic partitioning**.
- Total free space is enough but scattered in non-contiguous holes.

```
Memory:
┌──────────┐
│ Process A│  20MB
├──────────┤
│ FREE     │  5MB   ← Hole 1
├──────────┤
│ Process B│  15MB
├──────────┤
│ FREE     │  6MB   ← Hole 2
├──────────┤
│ Process C│  10MB
├──────────┤
│ FREE     │  4MB   ← Hole 3
└──────────┘
Total Free = 15MB but no process needing 12MB can fit!
```

## 14. Memory Allocation Strategies

When a new process needs memory, the OS must find a suitable free hole.

### 14.1 First Fit
- Allocate the **first hole** large enough to satisfy the request.
- Search starts from the beginning every time.
- **Fast** but may leave many small holes at the start.

```
Holes: 50K, 20K, 30K, 45K
Request: 25K
First Fit selects: 50K hole (first one large enough)
```

### 14.2 Next Fit
- Like First Fit but search starts where the **last allocation** ended.
- More uniform distribution of allocations.
- Avoids re-scanning already-checked areas.

### 14.3 Best Fit
- Allocate the **smallest hole** that is large enough.
- Minimizes wasted space per allocation.
- **Slow** — must scan entire list.
- Produces many tiny leftover holes (may be unusable).

```
Holes: 50K, 20K, 30K, 45K
Request: 25K
Best Fit selects: 30K hole (smallest sufficient hole)
Leftover: 5K (small, often unusable)
```

### 14.4 Worst Fit
- Allocate the **largest available hole**.
- Produces the largest leftover hole (more likely to be reusable).
- **Slow** — must scan entire list.
- Counterintuitively, sometimes better than Best Fit for future allocations.

```
Holes: 50K, 20K, 30K, 45K
Request: 25K
Worst Fit selects: 50K hole (largest)
Leftover: 25K (still usable)
```

### 14.5 Quick Fit
- Maintains **separate lists** for common sizes (e.g., 4KB list, 8KB list, 16KB list).
- Very fast allocation — just take from the right list.
- **Problem:** Coalescing (merging free holes) is expensive.

**Comparison:**

| Strategy | Speed | Fragmentation | Notes |
|---|---|---|---|
| First Fit | Fast | Moderate | Good in practice |
| Next Fit | Fast | Moderate | More uniform |
| Best Fit | Slow | Worst (tiny holes) | Minimizes waste per request |
| Worst Fit | Slow | Better | Largest leftover reusable |
| Quick Fit | Fastest | Moderate | Coalescing expensive |

## 15. Coalescing and Compaction

### 15.1 Coalescing (Merging Adjacent Holes)
When a process is freed, **adjacent free holes are merged** into one larger hole.

```
Before Process B exits:
┌──────────┐
│ Process A│
├──────────┤
│ FREE     │  10K
├──────────┤
│ Process B│  ← exits
├──────────┤
│ FREE     │  15K
├──────────┤
│ Process C│
└──────────┘

After coalescing when B exits:
┌──────────┐
│ Process A│
├──────────┤
│          │
│ FREE     │  10K + B_size + 15K = one large hole
│          │
├──────────┤
│ Process C│
└──────────┘
```

### 15.2 Compaction
Moving all processes to one end of memory so all free space is consolidated.

```
Before:                    After Compaction:
┌──────────┐               ┌──────────┐
│ Process A│               │ Process A│
├──────────┤               ├──────────┤
│ FREE     │               │ Process B│
├──────────┤               ├──────────┤
│ Process B│               │ Process C│
├──────────┤               ├──────────┤
│ FREE     │               │          │
├──────────┤               │  FREE    │  ← Large contiguous hole
│ Process C│               │          │
├──────────┤               │          │
│ FREE     │               └──────────┘
└──────────┘
```

**Problem with Compaction:**
- Requires moving processes in memory (expensive — copying data).
- All addresses must be updated.
- Processes must be stopped during compaction.
- High overhead — used only as a last resort.

## 16. Virtual Memory

**Virtual Memory** allows processes to use more memory than physically available in RAM by using disk space as an extension of RAM.

```
Process sees:           Reality:
┌──────────┐            ┌──────────┐         ┌────────────────┐
│  4GB     │            │   RAM    │         │   Disk (Swap)  │
│  address │            │  2GB     │◄──────► │   2GB          │
│  space   │            │  (part   │         │  (rest of      │
│  (all    │            │  of      │         │   process)     │
│  mine!)  │            │  process)│         │                │
└──────────┘            └──────────┘         └────────────────┘
```

**Key idea:** The process thinks it has a large contiguous address space. The OS manages which parts are actually in RAM vs disk, transparently.

**Benefits:**
- Programs larger than RAM can run.
- More processes can run concurrently.
- Processes are isolated from each other.

### 16.1 Limitations of Virtual Memory
- Performance overhead — disk access is thousands of times slower than RAM.
- **Thrashing** can occur (excessive page swapping).
- Requires hardware support (MMU).
- Complexity in OS design.

### 16.2 Virtual Address Space vs Physical Address Space

| | Virtual Address Space | Physical Address Space |
|---|---|---|
| What it is | What the process sees | Actual RAM available |
| Size | Can be very large (e.g., 4GB on 32-bit, 256TB on 64-bit) | Limited by installed RAM |
| Managed by | OS + MMU | Hardware (RAM chips) |
| Contents | May include unmapped regions | Only holds currently needed pages |

## 17. Paging

**Paging** divides both virtual memory and physical memory into fixed-size blocks:
- **Pages** — fixed-size blocks of virtual address space.
- **Frames** — fixed-size blocks of physical memory (same size as pages).
- Typical page size: 4KB.

```
Virtual Memory (Pages):          Physical Memory (Frames):
┌──────────────┐                 ┌──────────────┐
│   Page 0     │ ──mapping──►    │   Frame 3    │
├──────────────┤                 ├──────────────┤
│   Page 1     │ ──mapping──►    │   Frame 7    │
├──────────────┤                 ├──────────────┤
│   Page 2     │ ──mapping──►    │   Frame 1    │
├──────────────┤                 ├──────────────┤
│   Page 3     │ ──on disk──►    │   Frame 5    │
└──────────────┘                 └──────────────┘
```

**Address Translation:**
```
Virtual Address = Page Number + Offset
Physical Address = Frame Number (from Page Table) + Offset

Example:
  Page size = 4KB = 4096 bytes
  Virtual Address = 8192 + 100 → Page 2, Offset 100
  Page Table says: Page 2 → Frame 5
  Physical Address = 5 × 4096 + 100 = 20580
```

**Advantages of Paging:**
- No external fragmentation — frames can be anywhere in RAM.
- Supports virtual memory easily.
- Simple allocation — just find any free frame.

**Disadvantages:**
- Internal fragmentation (last page may not be completely filled).
- Page table overhead — needs memory itself.

---

## 18. Page Table

The **page table** is a data structure maintained by the OS for each process. It maps **page numbers to frame numbers**.

```
Page Table for Process P:
┌──────────┬─────────────┬──────────────┐
│  Page #  │  Frame #    │  Flags       │
├──────────┼─────────────┼──────────────┤
│    0     │     3       │  Present=1   │
│    1     │     7       │  Present=1   │
│    2     │     1       │  Present=1   │
│    3     │     -       │  Present=0   │  (on disk)
│    4     │     5       │  Present=1   │
└──────────┴─────────────┴──────────────┘

Flags:
  Present/Valid bit: 1 = in RAM, 0 = on disk
  Dirty bit: 1 = page modified, needs to be written to disk
  Referenced bit: 1 = recently accessed
  Protection bits: Read/Write/Execute permissions
```

---

## 19. Demand Paging

**Demand Paging** = pages are loaded into RAM **only when needed** (on demand), not all at once.

```
Process starts:           Page accessed:         Page not in RAM:
┌────────────┐            ┌────────────┐          ┌────────────┐
│ Page 0     │ in RAM     │ CPU access │          │ PAGE FAULT!│
│ (rest on   │            │ Page 3     │──────►   │ Load from  │
│  disk)     │            └────────────┘          │ disk       │
└────────────┘                                    └────────────┘
```

**Benefits:**
- Less RAM needed to start a process.
- Faster process startup.
- Only frequently used pages stay in RAM.

---

## 20. Page Faults

A **page fault** occurs when a process accesses a page that is **not currently in RAM** (present bit = 0).

### Handling Page Faults:

```
Step 1: CPU tries to access Page P → not in RAM → PAGE FAULT exception
Step 2: OS traps the fault (interrupt handler)
Step 3: OS finds the page on disk (in swap space)
Step 4: Find a free frame in RAM (or evict a page if RAM is full)
Step 5: Load the page from disk into the free frame
Step 6: Update the page table (set Present bit = 1, record frame number)
Step 7: Restart the instruction that caused the fault
Step 8: Process continues normally
```

**Page Fault Rate affects performance significantly:**
- High page fault rate → process spends most time waiting for disk → slow
- Low page fault rate → most accesses are to RAM → fast

## 21. Translation Look-aside Buffer (TLB)

The **TLB** is a **hardware cache** for recent page table entries. It dramatically speeds up address translation.

**Problem without TLB:** Every memory access requires 2 memory reads:
1. Read page table to find frame number.
2. Read actual data from that frame.

**With TLB:**
```
Virtual Address → Check TLB
                      │
                ┌─────┴──────┐
                │            │
           TLB Hit        TLB Miss
           (fast!)        (slow — must read page table)
                │            │
                ▼            ▼
           Frame # found   Read page table, update TLB
                │
                ▼
           Access physical memory
```

**TLB Hit Ratio:** Typically 95–99%. This makes paging nearly as fast as no paging.

**Effective Access Time (EAT):**
```
EAT = h × (TLB access time + memory access time)
    + (1-h) × (TLB access time + 2 × memory access time)

Where h = TLB hit ratio

Example: TLB access = 10ns, Memory = 100ns, h = 0.95
EAT = 0.95 × (10+100) + 0.05 × (10+200)
    = 0.95 × 110 + 0.05 × 210
    = 104.5 + 10.5 = 115 ns
```

## 22. Page Replacement Algorithms

When RAM is full and a new page must be loaded, the OS must **evict** an existing page. The algorithm decides which page to remove.

**Goal:** Minimize the number of page faults.

---

### 22.1 FIFO (First In First Out)

**Rule:** Replace the page that has been in memory the **longest** (oldest page).

**Example:**
```
Reference string: 7 0 1 2 0 3 0 4 2 3 0 3 2 1 2 0 1 7 0 1
Frames: 3

Reference  Frames          Page Fault?
7          7  -  -         YES (miss)
0          7  0  -         YES (miss)
1          7  0  1         YES (miss)
2          2  0  1         YES (miss, evict 7 — oldest)
0          2  0  1         NO  (hit)
3          2  3  1         YES (miss, evict 0 — oldest)
0          2  3  0         YES (miss, evict 1 — oldest)
4          4  3  0         YES (miss, evict 2 — oldest)
2          4  2  0         YES (miss, evict 3 — oldest)
3          4  2  3         YES (miss, evict 0)
0          0  2  3         YES (miss, evict 4)
3          0  2  3         NO  (hit)
2          0  2  3         NO  (hit)
1          1  2  3         YES (miss, evict 0)
2          1  2  3         NO  (hit)
0          1  0  3         YES (miss, evict 2)
1          1  0  3         NO  (hit)
7          7  0  3         YES (miss, evict 1)
0          7  0  3         NO  (hit)
1          7  0  1         YES (miss, evict 3)

Total Page Faults: 15
```

**Belady's Anomaly:**
- **Unexpected:** Adding more frames can INCREASE page faults with FIFO.
- **Example:** With 3 frames = 9 faults, with 4 frames = 10 faults!
- This is called **Belady's Anomaly** — unique to FIFO.
- Optimal and LRU do NOT suffer from Belady's Anomaly.

---

### 22.2 Optimal Page Replacement

**Rule:** Replace the page that will **not be used for the longest time** in the future.

**Example:**
```
Reference string: 7 0 1 2 0 3 0 4 2 3 0 3 2 1 2 0 1 7 0 1
Frames: 3

Reference  Frames          Action
7          7  -  -         Page fault
0          7  0  -         Page fault
1          7  0  1         Page fault
2          2  0  1         Page fault (evict 7 — used farthest in future)
0          2  0  1         Hit
3          2  0  3         Page fault (evict 1 — used farthest)
0          2  0  3         Hit
4          4  0  3         Page fault (evict 2)
2          4  0  2         Page fault (evict 3)

Total Page Faults: 9 (less than FIFO!)
```

**Advantages:** Minimum possible page faults — optimal benchmark.
**Disadvantages:** **Cannot be implemented in practice** — OS cannot know future reference string. Used only as a theoretical benchmark to compare other algorithms.

---

### 22.3 LRU (Least Recently Used)

**Rule:** Replace the page that was **last used longest ago** (least recently used).

**Rationale:** Pages used recently will likely be used again soon (locality of reference). Pages not used recently are less likely to be needed.

**Example:**
```
Reference string: 7 0 1 2 0 3 0 4 2 3 0 3 2 1 2 0 1 7 0 1
Frames: 3

Reference  Frames              Replaced    Fault?
7          [7]                 —           YES
0          [7,0]               —           YES
1          [7,0,1]             —           YES
2          [0,1,2]             7 (LRU)     YES
0          [1,2,0]             —           NO  (hit, 0 becomes MRU)
3          [2,0,3]             1 (LRU)     YES
0          [2,3,0]             —           NO
4          [3,0,4]             2 (LRU)     YES
2          [0,4,2]             3 (LRU)     YES
3          [4,2,3]             0 (LRU)     YES
0          [2,3,0]             4 (LRU)     YES
3          [2,3,0]             —           NO
2          [3,0,2]             —           NO
1          [0,2,1]             3 (LRU)     YES
2          [0,1,2]             —           NO
0          [1,2,0]             —           NO
1          [2,0,1]             —           NO
7          [0,1,7]             2 (LRU)     YES
0          [1,7,0]             —           NO
1          [7,0,1]             —           NO

Total Page Faults: 12
```

**Implementation:**
- **Counter-based:** Each page has a timestamp of last use. Evict page with smallest timestamp.
- **Stack-based:** Keep pages in a stack; on access, move page to top. Bottom = LRU page.

**LRU is widely used** — good approximation of optimal without requiring future knowledge.

---

### 22.4 Second Chance Algorithm (Clock Algorithm)

**Rule:** FIFO with a second chance — before evicting a page, check if it has been recently referenced.

**How it works:**
- Each page has a **reference bit** (0 or 1).
- When page is accessed → set reference bit to 1.
- On page fault, scan pages in FIFO order:
  - If reference bit = 0 → EVICT this page.
  - If reference bit = 1 → Give second chance: set bit to 0, move to back of queue, check next page.

```
Pages in FIFO order with reference bits:
[Page A: bit=1] → [Page B: bit=0] → [Page C: bit=1] → [Page D: bit=0]
                       ↑
              Evict Page B (bit=0)
```

**Better than pure FIFO** because recently used pages get a second chance.

## 23. Concept of Locality of Reference

Processes tend to access a **small subset of pages** at any given time. This is called the **principle of locality**.

**Two types:**

**Temporal Locality:**
- Recently accessed data is likely to be accessed again soon.
- **Example:** Loop variable — accessed on every iteration.

**Spatial Locality:**
- Data near recently accessed data is likely to be accessed soon.
- **Example:** Array traversal — if element[5] is accessed, element[6] will soon be accessed.

```
Program accessing array:
for(i=0; i<1000; i++) {
    sum += a[i];        // Spatial locality: a[0],a[1],a[2]...
}                       // Temporal locality: sum, i accessed repeatedly
```

**Why it matters:** Locality of reference is why virtual memory with paging works well in practice. The working set of pages is usually small and stable for periods of time.

## 24. Thrashing

**Thrashing** occurs when a process spends **more time swapping pages** than actually executing.

```
Timeline without thrashing:
[Execute] [Execute] [Execute] [Execute]   ← CPU busy

Timeline with thrashing:
[Page Fault][Swap][Page Fault][Swap][Page Fault][Swap]   ← CPU idle, I/O busy
```

**Cause:** Too many processes in memory, each gets too few frames. Every page access causes a fault.

**Detection:** CPU utilization drops sharply even though many processes are runnable.

**Solutions:**
1. **Reduce degree of multiprogramming** — fewer processes in memory.
2. **Increase RAM.**
3. **Use Working Set model** to allocate enough frames.
4. **Page Fault Frequency (PFF)** control — if PFF too high, give more frames.

---

## 25. Working Set

The **working set** of a process is the set of pages it has **actively used** in the recent past (within a time window Δ).

```
Working Set W(t, Δ) = set of pages referenced in the interval (t-Δ, t)

If Δ = 10 time units:
References: 2 5 2 3 5 6 2 3 7 3 | 6 2 3 ...
                                   ← Δ →
Working Set at time t = {6, 2, 3, 7}  (4 distinct pages)
```

**Working Set Model:**
- Each process is allocated frames equal to its working set size.
- If total working set sizes > total frames → reduce multiprogramming (suspend a process).
- Prevents thrashing.

## 26. Segmentation

**Segmentation** divides a program into **logical segments** based on logical structure (code, data, stack, heap) rather than fixed-size pages.

```
Program divided into segments:
┌──────────────────┐  Segment 0: Code (text)
│   int main() {  │
│     ...          │
└──────────────────┘
┌──────────────────┐  Segment 1: Data
│   int x = 5;    │
│   char str[];   │
└──────────────────┘
┌──────────────────┐  Segment 2: Stack
│   function calls │
│   local vars     │
└──────────────────┘
┌──────────────────┐  Segment 3: Heap
│   malloc'd memory│
└──────────────────┘
```

**Segment Table:**
```
┌────────────┬─────────┬────────┐
│ Segment #  │  Base   │ Limit  │
├────────────┼─────────┼────────┤
│     0      │  4000   │  600   │  Code at physical 4000, size 600
│     1      │  6000   │  500   │  Data at physical 6000, size 500
│     2      │  8000   │  200   │  Stack at physical 8000, size 200
└────────────┴─────────┴────────┘
```

**Address Translation:**
```
Logical Address = (Segment #, Offset)
Physical Address = Base[Segment#] + Offset

Check: Offset < Limit[Segment#]  → valid
       Offset ≥ Limit[Segment#]  → Segmentation Fault!
```

### 26.1 Need for Segmentation
- **Logical structure** — different parts of program have different sizes and protection.
- **Sharing** — code segment can be shared between processes (e.g., shared libraries).
- **Protection** — code segment: read+execute only; data segment: read+write.
- **Dynamic growth** — stack and heap can grow independently.

---

## 27. Segmentation with Paging (MULTICS)

**Problem with pure segmentation:** Variable-size segments → external fragmentation.
**Problem with pure paging:** No logical structure → hard to share/protect.

**Solution: Combine both!** — Used in **MULTICS** (Multiplexed Information and Computing Service).

```
Logical Address → Segment Table → Page Table → Physical Address

Process:
1. CPU generates (Segment#, Page#, Offset)
2. Segment table finds the page table for that segment
3. Page table translates Page# → Frame#
4. Physical Address = Frame# + Offset
```

**Benefits:**
- Each segment has its own page table.
- No external fragmentation (pages are fixed size).
- Logical organization preserved (segments).
- Segments can grow dynamically (just add more pages).
- Sharing at segment level possible.

**Modern OS:** Intel x86 architecture supports both. Linux uses a flat segmentation model (all segments start at 0) and relies mainly on paging.

> **Key formulas:**
> ```
> Physical Address = Frame Number × Page Size + Offset
> Page Number      = Logical Address / Page Size
> Offset           = Logical Address mod Page Size
> CPU Utilization  = 1 - p^n  (multiprogramming model)
> EAT (with TLB)   = h(t_TLB + t_mem) + (1-h)(t_TLB + 2×t_mem)
> ```

> **Golden rule for page replacement numericals:** Always track which pages are currently in frames, mark hits clearly, and count only misses as page faults. For LRU, always track when each page was last used.
