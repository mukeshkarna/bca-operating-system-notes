# Chapter 5: Memory Management

## Introduction
Memory management is the OS function that tracks every byte of memory, allocates it to processes, and reclaims it when a process finishes.
- Goals: efficient utilization, **protection** (isolating processes), **relocation** (running at different addresses), sharing, logical organization.
- Without it, one buggy program could crash or corrupt every other running program.
- **Real-world example:** When Chrome, Word, and Spotify run together, the OS gives each its own memory space so Chrome can never overwrite Word's data — and reclaims that memory instantly when you close the app.

## Memory Hierarchy
Storage tiers ranked by speed/cost (fastest+priciest at top) vs. size/cheapness (largest+cheapest at bottom):

| Tier | Real-world example |
|---|---|
| Registers | CPU registers (~64 bytes) — hold the value the ALU is computing right now |
| Cache Memory | L1/L2/L3 CPU cache (8-32 MB) — recently used instructions/data |
| Main Memory (RAM) | 8-32 GB DDR RAM in a laptop — actively running programs |
| Magnetic Disks | 512 GB-2 TB SSD/HDD — files, OS, installed apps |
| Magnetic Tape | Enterprise LTO tape archives — banks/data centers store decades of backups cheaply |

## Logical vs. Physical Address Space
- **Logical (virtual) address** — generated by the CPU while a program runs.
- **Physical address** — the actual RAM location, generated by the MMU at run time.
- Compile-time/load-time binding: logical = physical. Execution-time (dynamic) binding lets them differ, enabling swapping/relocation.
- **Real-world example:** every Chrome tab thinks it owns address 0x400000; the MMU maps each to a different physical location so tabs never collide.

## Memory Management with Swapping
Moves an entire process to disk (swap-out) and back (swap-in) so more processes can share limited RAM. Swapped-in processes may land at a new address, requiring relocation.
- **Real-world example:** Windows' pagefile.sys / Linux-macOS swap space moves idle apps to disk when RAM is full.

## Memory Management with Bitmaps
Memory divided into fixed-size units; one bit per unit (0=free, 1=occupied). Smaller units = finer tracking but bigger bitmap.
- **Real-world example:** SSD controllers track free vs. written blocks with bitmaps, like a library seating chart.

## Memory Management with Linked Lists
Free/used segments tracked as a linked list (P=process, H=hole) with start address + length. Adjacent holes merge (coalescing) when a process exits.
- **Real-world example:** C's malloc/free maintain exactly this kind of list.

## Memory Management without Swapping
Simplest scheme — one process in memory at a time alongside the OS, no swapping/partitioning needed.
- **Real-world example:** early MS-DOS PCs, simple thermostat firmware.

## Mono-programming
Only one user program resides in memory; CPU idles completely during I/O waits.
- **Real-world example:** an ATM machine or ticket kiosk.

## Multi-programming — Dedicated Partitions (Absolute Translation)
Memory split into fixed partitions, each with its own job queue; programs compiled for one specific partition's address range.
- **Real-world example:** IBM OS/360 (MFT) batch jobs assigned to fixed partitions, like reserved parking spots.

## Multi-programming — Single Queue (Re-locatable Translation)
All jobs share one queue; whenever a partition frees up, the best-fitting job loads using relative addressing.
- **Real-world example:** bank "take-a-number" system routes the next customer to any free teller.

## Mono- vs. Multi-programming

| Aspect | Mono-programming | Multi-programming |
|---|---|---|
| Processes in memory | 1 | Multiple |
| CPU utilization | Low (idle during I/O) | High |
| Complexity | Very simple | Needs partitioning, relocation, protection |
| Throughput | Low | Higher |
| Example | ATM, ticket kiosk | Windows/macOS/Linux multitasking |

## Modeling Multiprogramming
If a process waits on I/O for fraction *p* of its time, and *n* processes share memory: **CPU utilization = 1 − pⁿ**. More resident processes push utilization toward 100%.
- **Real-world example:** a web server serves other requests while one waits on database I/O, instead of leaving the CPU idle.

## Fixed (Static) Partitioning
Memory split into a fixed number of partitions at boot; never resized. Simple, but wastes space when jobs are smaller than their partition (internal fragmentation).
- **Real-world example:** a hotel with only 3 fixed room sizes — a solo traveler in a suite wastes space.

## Variable (Dynamic) Partitioning
Partitions created on demand, exactly sized to each process. Leads to scattered free holes (external fragmentation) over time.
- **Real-world example:** a moving truck custom-cut for each box — efficient at first, oddly shaped gaps later.

## Relocation
Maps a program's logical addresses to different physical addresses each time it's loaded/swapped in, via a hardware **relocation (base) register**.
- **Real-world example:** mail forwarding — your address never changes, but the postal service redirects mail to your new home.

## Protection
Base + limit registers check every address a process generates against its partition range; violations trigger a hardware trap.
- **Real-world example:** why a buggy Word doc crashes only Word — a "segmentation fault" is protection hardware catching an out-of-bounds access.

## Internal Fragmentation
Wasted space *inside* an allocated block when the process doesn't use all of it. Common with fixed partitioning/paging.
- **Real-world example:** a 12-egg carton when you only need 10 eggs.

## External Fragmentation
Free memory scattered into small non-contiguous holes; no single hole is big enough even though total free space is large.
- **Real-world example:** a parking lot with enough total empty spots for a bus, but scattered as single-car gaps.

## Allocation Strategies

| Strategy | Speed | Result |
|---|---|---|
| **First Fit** | Fast | Allocates first hole big enough; good overall |
| **Next Fit** | Fast | Resumes from last position; slightly worse than First Fit in practice |
| **Best Fit** | Slow (full scan) | Smallest sufficient hole; creates many tiny unusable fragments |
| **Worst Fit** | Slow (full scan) | Largest hole; exhausts big holes over time |
| **Quick Fit** | Very fast | Separate free-lists by size; coalescing is harder |


Real-world analogies: First Fit = grabbing the nearest parking spot; Best Fit = renting the smallest storage unit that fits; Worst Fit = always taking the biggest warehouse bay; Quick Fit = a hardware store's labeled screw bins.

## Coalescing and Compaction
- **Coalescing:** merges adjacent free holes into one when a process exits.
- **Compaction:** shuffles all processes together to combine every scattered hole into one large block.
- **Problem with compaction:** expensive data copying + every relocated process's base register must update; processes pause during the operation.
- **Real-world example:** Windows disk defragmentation — slides file fragments together, at the cost of time.

## Virtual Memory and Its Limitations
Lets a process run with an address space larger than physical RAM by keeping only active parts in memory (rest on disk), via paging/segmentation + MMU.
- **Limitations:** disk is far slower than RAM (thrashing risk), plus translation overhead (page tables, TLB).
- **Real-world example:** editing 4K video in Premiere Pro on an 8GB laptop works via virtual memory, but slows down (thrashing) if too little RAM is free.

## Chapter 5 Recap
Memory management balances utilization, protection, and speed — from simple mono-programming to virtual memory, every technique trades complexity for better CPU/RAM usage.
